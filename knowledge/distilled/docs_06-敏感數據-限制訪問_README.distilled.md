# 敏感憑證與配置管理知識文檔

**⚠️ 目錄警告**  
此文檔及相關目錄包含高度敏感的憑證、密鑰和配置資訊，包括 API 金鑰、服務器配置以及 Slack 2FA 備份碼。**嚴禁公開分享或上傳至任何非加密環境**。未經授權存取可能導致帳戶入侵、資料外洩或財務損失。

## 1. 文檔內容概述
此目錄儲存關鍵的安全憑證和恢復資源，用於系統維護和緊急恢復。核心檔案列表如下：

| 檔案名稱 | 描述 | 敏感度 |
|----------|------|--------|
| `14-info-resource-library.md` | API Keys & 服務器配置（包含第三方服務 API 金鑰、資料庫連接字串、雲端服務憑證等） | **極高**（直接暴露系統存取權） |
| `17-slack-recovery-codes.md` | Slack 2FA 備份碼（共 10 組一次性使用碼，用於雙因素認證恢復） | **高**（洩露將導致 Slack 帳戶完全控制） |

**脈絡說明**：  
- 這些檔案是從內部資源庫匯出，用於開發、DevOps 或緊急應變情境。  
- API 金鑰通常連結至 AWS、Google Cloud、Stripe 或其他 SaaS 服務，暴露將允許未授權計費或資料存取。  
- Slack 備份碼為標準 2FA 機制的一部分，每組碼僅限使用一次，遺失或洩露需立即重置 2FA。

## 2. 安全最佳實務與建議
為防止洩露，必須嚴格遵循以下指南：

### 立即行動項目
- **禁止推送至公開倉庫**：絕對不要將此目錄內容 commit 至 GitHub、GitLab 等公開或半公開 repo。即使私有 repo 也建議排除（使用 `.gitignore`）。
- **加密存儲**：  
  - 優先使用密碼管理器：**1Password** 或 **Bitwarden**（支援團隊共享與自動填入）。  
  - 替代方案：VeraCrypt 加密容器或 AWS S3 以 KMS 加密。
- **定期輪換**：  
  | 項目 | 輪換頻率 | 責任人 |
  |------|----------|--------|
  | API 金鑰 | 每 90 天或事件後 | DevOps 團隊 |
  | Slack 2FA 碼 | 使用後立即重置 | 帳戶擁有者 |
  | 服務器配置 | 每 6 個月 | 系統管理員 |

### 訪問控制
- **最小權限原則**：僅授權必要人員（e.g., 透過 RBAC 或 IAM 角色）。  
- **稽核記錄**：啟用所有存取日誌，定期審核（e.g., 使用 Splunk 或 ELK Stack）。  
- **備份策略**：僅儲存加密副本，避免明文備份。

## 3. 導入 Notion 前注意事項
若計劃將資源遷移至 Notion（或其他雲端筆記工具），**務必先處理敏感數據**：

1. **刪除或加密**：  
   - 手動移除或替換所有憑證為佔位符（e.g., `API_KEY: [已加密]`）。  
   - 使用 Notion 的加密區塊或外部連結至 1Password。
2. **驗證步驟**：  
   ```
   1. 掃描目錄：grep -r "sk_" . 或類似模式匹配 API 金鑰。
   2. 加密敏感檔案：ansible-vault encrypt 14-info-resource-library.md
   3. 測試導入：僅上傳清理後版本。
   4. 確認無洩露：使用工具如 TruffleHog 掃描。
   ```
3. **風險**：Notion 共享頁面易意外公開，優先考慮企業��� + 權限鎖定。

## 4. 緊急應變程序
- **偵測洩露**：監控 GitHub 搜尋或 Have I Been Pwned。  
- **回應流程**：  
  1. 立即輪換所有受影響金鑰。  
  2. 通知受影響服務（e.g., Slack admin）。  
  3. 進行安全稽核。  
- **工具推薦**：  
  | 工具 | 用途 |
  |------|------|
  | `git-secrets` | 防止 commit 敏感數據 |
  | `detect-secrets` | 掃描程式碼洩露 |
  | `gitleaks` | CI/CD 整合檢查 |
